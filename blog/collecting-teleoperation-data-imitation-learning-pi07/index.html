Collecting Teleoperation Data for Imitation Learning on a Consumer GPU with π0.7

June 9, 2026 · Prometheus Robotics

Teleoperation data is the fuel of modern robot learning — but what you burn it in matters. Train a policy from scratch and it only ever knows the exact task you demonstrated. Fine-tune a pretrained vision-language-action (VLA) model on the same data and you inherit a generalist that already understands objects, scenes, and language — then specialize it to your robot in an afternoon.

π0.7 is the latest in the π (“pi”) series of VLA models from Physical Intelligence, and it is built for exactly this workflow: collect demonstrations, fine-tune, deploy. This guide covers collecting that data on the Prometheus humanoid and fine-tuning π0.7 on a single consumer GPU.

New to imitation learning? Start with the companion guide, collecting teleoperation data for ACT. It covers the data-collection basics from scratch — and the dataset you record there feeds π0.7 directly.

What π0.7 is

The π series pairs a vision-language model (VLM) backbone with an action expert that generates continuous, high-frequency action chunks via flow matching. The model is pretrained on a large, diverse corpus of robot interaction data, so out of the box it already maps images and a natural-language instruction to plausible motion across many embodiments.

Each release sharpened that recipe: π0 established the flow-matching VLA, π0.5 pushed open-world generalization to unseen environments, and π0.7 is the current checkpoint in that lineage. For your purposes the key property is the same throughout: you don’t train it from zero — you fine-tune a capable generalist on a few hours of your own demonstrations.

Why fine-tune a VLA instead of training ACT?

  • Generalization. Because π0.7 starts from broad pretraining, it transfers to new object positions, lighting, and even novel objects far better than a policy trained only on your 50 demos.
  • Language conditioning. You can tell it what to do (“put the blue cube in the bin”), and a single model can cover several related tasks.
  • Fewer demos per task. Pretraining does the heavy lifting, so each new skill needs less data than learning from scratch.

The trade-off is size: π0.7 is a large model, so fine-tuning costs more than ACT. The practical workaround is parameter-efficient fine-tuning (LoRA), which brings the memory footprint down to roughly what a single 24 GB consumer GPU (an RTX 4090, say) provides.

Collecting the data on Prometheus

The collection loop is the same one described in the ACT guide — Prometheus ships with the teleoperation pipeline ready, so you record without integration work:

  • VR teleop with a Meta Quest 3S, or the leader–follower / manual controller for precise trajectories.
  • Observations: the head-mounted stereo pair plus wrist cameras. The adjustable wrist-camera angle was designed with VLA models like the π series in mind — set it once and keep it fixed across demos.
  • Proprioception and actions: joint states and the target commands you teleoperate, recorded in the LeRobot dataset format.

Annotate with language. The one thing a VLA needs that ACT does not is an instruction per episode. Label each demonstration with the natural-language task it shows (“pick up the cup and place it on the saucer”). Consistent, descriptive labels are what unlock π0.7’s language conditioning at run time.

# Illustrative — record language-annotated teleop episodes
python -m lerobot.record \
    --robot.type=prometheus \
    --teleop.type=quest3s \
    --dataset.repo_id=yourlab/kitchen_tasks \
    --dataset.single_task="put the blue cube in the bin" \
    --dataset.num_episodes=40 \
    --dataset.fps=30

Fine-tuning π0.7 on a single GPU

With demonstrations in hand, fine-tuning is a LoRA run against the released π0.7 checkpoint. Convert (or point at) your LeRobot dataset, then launch:

# Illustrative — LoRA fine-tune of pi0.7 on your dataset
python -m openpi.train pi07_lora \
    --dataset.repo_id=yourlab/kitchen_tasks \
    --lora=true \
    --batch_size=4 \
    --num_steps=30000 \
    --output_dir=outputs/pi07_kitchen

LoRA keeps the trainable parameters and optimizer state small enough to fit on one 24 GB card; full fine-tuning is heavier and is where a rented A100/H100 still helps. For most labs validating a task, the consumer-GPU LoRA path is enough to get a working, language-conditioned policy.

Deploying π0.7 back on the robot

At inference the policy takes the live camera streams, the robot state, and a text instruction, and emits action chunks. Run it on a tethered workstation GPU and stream commands to Prometheus through the SDK’s REST API, or distil/quantize for the onboard Jetson. Because you can validate against the bundled simulator and the included URDF first, you catch issues before they touch hardware.

The practical recipe

  1. Record 30–50 language-annotated teleop demos per task on Prometheus (VR or leader–follower).
  2. Store them in LeRobot format — the same dataset works for ACT and π0.7.
  3. LoRA fine-tune π0.7 on a single 24 GB GPU.
  4. Validate in sim, then deploy via the SDK.

Start with ACT to get a feel for the data, then move to π0.7 when you need a policy that generalizes and listens to instructions. Same robot, same demonstrations — a far more capable result.

Model names and capabilities follow Physical Intelligence’s π-series releases; check the current openpi documentation for exact checkpoints, commands, and hardware requirements.
